# Toolset and Configuration Rules

## Toolset Files — Source of Truth

`config/toolsets/*.txt` define which DCT API endpoints belong to each persona toolset. These are the primary place to add new capabilities — no Python code changes needed.

**Format:**
```
# Toolset Name - N Tools
# Description: <shown in auto-mode discovery>
# Target users: <shown in auto-mode discovery>

# TOOL N: tool_name - Tool Description
METHOD|/endpoint/path/{pathParam}|action_name
```

After modifying a `.txt` file:
- Ensure the `action_name` is handled in the corresponding `tools/*_endpoints_tool.py`
- If adding a new `tool_name`, register it in `config/loader.py:get_modules_for_toolset()` inside `TOOL_TO_MODULE`

## Generated Code — Never Edit Directly

Tool modules written to `$TEMP/dct_mcp_tools/` are auto-generated by `tool_factory.py` from the DCT OpenAPI spec. Do not edit them — they are overwritten at startup.

## Adding a New Toolset

1. Create `config/toolsets/my_toolset.txt` following the format above
2. Add `TOOL_TO_MODULE` entries in `config/loader.py` for any new tool names introduced
3. No other code changes needed — the loader discovers `.txt` files automatically

## Adding a New Tool Module

1. Create `tools/my_resource_endpoints_tool.py` with a `register_tools(app, dct_client)` function
2. Add action handling matching the action names in the toolset `.txt` files
3. Register the tool name → module mapping in `config/loader.py:TOOL_TO_MODULE`
4. Apply `@log_tool_execution` to every tool function

## Confirmation Rules

`config/mappings/manual_confirmation.txt` controls which operations require confirmation.

**Format:**
```
METHOD|path_pattern|confirmation_level|message_template
```

| Level | When to use |
|-------|-------------|
| `standard` | Reversible but impactful operations |
| `elevated` | Hard-to-reverse operations |
| `manual` | Destructive/irreversible operations |
| `retention_check:N` | Conditional — triggers only if retention < N days |
| `policy_impact_check:N` | Conditional — triggers if policy affects > N objects |

- Use `*` as method wildcard
- Path patterns support `{paramName}` placeholders
- Rules are evaluated in order — first match wins

## Toolset Inheritance

Use `@inherit:parent_toolset_name` in a `.txt` file to include all APIs from another toolset:

```
@inherit:self_service
# Additional tools below...
```

The parent file must exist in `config/toolsets/`.

## Config Cache

`loader.py` caches parsed `.txt` files with `@lru_cache`. If toolset files change at runtime, call `config.loader.clear_cache()` to invalidate.
